## 2. 总体评价

计划覆盖了 PRD 和实施方案中最关键的 MVP 目标：FastMCP 骨架、OAuth 双模式、token 加密、SQLite、本地缓存、能力探测、降级报告、统一错误和安全边界。对“当前无源码、无 Git 仓库、无真实抖音凭证”的现状判断准确，实施范围也没有越过只读和官方 OpenAPI 边界。

最大风险是 HTTP 模式访问控制与 FastMCP 具体版本的集成方式可能存在不确定性。计划已经通过启动守卫、ASGI 中间件和反向代理认证说明做了兜底，但实施时需要诚实记录实际采用的方式。另一个风险是依赖安装受限，计划中已要求如实记录并按权限流程处理。

整体看，计划足够具体，文件清单、服务边界、测试策略和回滚方式明确，下游实施可以直接执行。

### P2（一般）

- **HTTP 访问控制实现依赖 FastMCP 版本细节**：计划提出 ASGI 中间件、FastMCP 原生 auth 和反向代理兜底，但没有在计划中锁定具体 FastMCP API。

  - 影响：实施时可能需要根据实际依赖版本调整启动方式。
  - 建议：实施日志中记录实际 FastMCP 版本和最终采用的 HTTP 防护方式；若无法在 MCP HTTP 层挂中间件，至少做到缺 API Key 拒绝 HTTP 启动并在文档中要求反向代理认证。

### P3（提示）

- **真实官方接口字段仍需后续确认**：计划依赖 `docs/api-mapping.md` 集中配置，但没有承诺字段完全最新。
  - 位置：风险与对策
  - 影响：真实上线前仍需用户在抖音开放平台后台核对。
  - 建议：实施时在 `docs/api-mapping.md` 标注“以官方后台最新文档为准”。

## 4. 改进建议汇总

1. 实施时优先保证核心 service/storage 层可离线测试，不要让 FastMCP 或网络依赖阻断所有测试。
2. HTTP 模式必须至少实现启动守卫：未配置访问控制时不允许启动 HTTP transport。
3. `docs/api-mapping.md` 中所有官方路径、scope、限制都要标注需要以抖音开放平台最新后台为准。
4. TokenStore 测试必须直接检查 SQLite 中不含明文 token。
5. 报告测试必须覆盖数据缺失不是表现差的文案边界。
